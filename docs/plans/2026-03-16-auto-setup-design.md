# AgentForge Auto-Setup 设计文档

## 目标

用户将 AgentForge 指向任意项目 repo（ML 训练、Triton 优化、回测因子挖掘等），框架自动分析项目结构并生成所需配置文件（challenge.yaml、benchmark.py、test_suite.py），用户逐个确认后即可启动优化循环。零手动配置。

## 架构

```
agentforge run /path/to/repo
  │  challenge.yaml 不存在 → 自动进入 setup
  │
  ├─ ProjectAnalyzer.analyze(workdir)
  │     Codex exec -s read-only 分析 repo → ProjectProfile(JSON)
  │
  ├─ ConfigGenerator.generate(profile)
  │     模板渲染 → challenge.yaml, benchmark.py, test_suite.py
  │
  ├─ InteractiveConfirm.confirm_each(files)
  │     逐个展示 → 用户 Y/n/edit
  │
  ├─ 写入文件 + git commit
  └─ 继续正常 Orchestrator 流程
```

### 设计原则

- **ProjectAnalyzer** 只做分析，输出 ProjectProfile 数据结构
- **ConfigGenerator** 只做模板渲染，输入 ProjectProfile 输出文件内容
- 两者完全解耦，可独立测试
- Codex 只做"理解"（read-only），不做"执行"

## 数据结构：ProjectProfile

```python
@dataclass
class ProjectProfile:
    # 元信息（通用，不假设项目类型）
    description: str           # Codex 对项目的描述

    # 入口
    run_command: str           # 主运行命令 "python train.py"
    run_args: list[str]        # 默认参数

    # 评估
    eval_metric: str           # 指标名 "val_loss" / "sharpe_ratio" / "kernel_throughput"
    eval_direction: str        # "minimize" | "maximize"
    eval_method: str           # Codex 描述如何获取指标
    baseline_value: float | None  # 当前基准值
    suggested_target: float    # 建议目标值

    # 文件分类
    writable: list[str]        # 可修改的文件/目录
    readonly: list[str]        # 不许动的文件/目录

    # 运行环境
    python_cmd: str            # "python3" / "conda run -n env python"
    needs_gpu: bool

    # benchmark 生成所需
    result_location: str       # "checkpoint" / "stdout" / "output_file"
    result_pattern: str        # glob 模式找结果文件
    metric_extraction: str     # Python 代码片段，提取指标到变量 score
    import_checks: str         # Python 代码片段，验证依赖可导入
```

## Codex 分析提示词

```
你是一个项目分析器。分析当前目录的代码，返回结构化 JSON。

你可以：
  - 读取任何文件（cat, head, find, ls）
  - 联网搜索相关文献/benchmark
  - 跑简短的测试（python -c "import torch"）
你不能：
  - 修改任何文件

请分析并将 JSON 写入 .agentforge/project_profile.json：
{完整 ProjectProfile schema}

重点分析：
1. 找到主运行入口和启动方式
2. 找到评估指标（看 eval/val/metric 相关代码）
3. 写一段 Python 代码片段放入 metric_extraction，
   该代码执行后将指标值赋给变量 score
4. 识别哪些文件是数据/评估（不该改），哪些是可优化的（可以改）
5. 根据项目现状建议合理的优化目标值
6. 写一段 import 检查代码放入 import_checks
```

## 模板

### challenge.yaml

```yaml
challenge:
  name: '{description[:60]}'
  description: |
    {description}
    Run command: {run_command} {run_args}
    Baseline: {baseline_value}
target:
  metric: {eval_metric}
  value: {suggested_target}
  direction: {eval_direction}
tests:
  smoke: '{python_cmd} test_suite.py'
  full: '{python_cmd} test_suite.py'
  benchmark: '{python_cmd} benchmark.py'
constraints:
  writable: {writable}
  read_only: {readonly}
```

## 交互确认流程

```
[AgentForge] 项目分析完成 (Codex, 1.5 min)

━━━ 1/3: challenge.yaml ━━━
(显示完整内容)
确认？[Y/n/edit] >

━━━ 2/3: benchmark.py ━━━
(显示完整内容)
确认？[Y/n/edit] >

━━━ 3/3: test_suite.py ━━━
(显示完整内容)
确认？[Y/n/edit] >

[AgentForge] 配置已保存并 commit。开始优化...
```

- `Y` / 回车：确认
- `n`：重新让 Codex 分析这部分
- `edit`：打开 $EDITOR 手动编辑

## 新增模块

| 文件 | 类 | 职责 |
|------|-----|------|
| `agentforge/analyzer.py` | `ProjectProfile` | 数据结构 |
| `agentforge/analyzer.py` | `ProjectAnalyzer` | 调用 Codex 分析 repo |
| `agentforge/generator.py` | `ConfigGenerator` | 模板渲染生成 3 个文件 |
| `agentforge/confirm.py` | `InteractiveConfirm` | 交互式逐步确认 |

## 对现有模块的修改

| 文件 | 修改 |
|------|------|
| `agentforge/orchestrator.py` | `_init_or_resume()` 开头加 auto-setup 检测 |
| `agentforge/cli.py` | `run` 命令允许 config_path 为空 |

## 不变的部分

Phase 1（Codex 开发策略）、Phase 2（并行训练）、评分、监控、状态管理等全部不变。Auto-setup 只影响初始化阶段。
